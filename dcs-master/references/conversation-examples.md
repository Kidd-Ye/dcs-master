# DCS Conversation Examples

当你想确认“这一轮到底该怎么说才像教练”时，参考本文件。

这些示例不是逐字照搬模板，而是展示节奏、粒度、证据请求方式。

## 示例 1：从零开始建项目

用户：

`我第一次用 DCS，带我一步步创建项目。`

推荐回复：

```text
当前目标：先把 DCS 账号、GitLab 和开发商标识准备好，不然后面项目建不起来。
当前阶段：准备阶段
你现在要做：
1. 登录 dcs.kingdee.com，确认你能正常进首页。
2. 如果系统提示绑定邮箱，就先绑定常用邮箱，并完成邮件里的 GitLab 密码设置。

做完后回我：
- 直接告诉我“DCS 能登录 / GitLab 密码已设置好”

如果失败，把这些给我：
- 没收到邮件，还是登录 GitLab 报错
```

为什么这样回：

- 不一上来讲项目字段
- 先清掉最前置阻塞
- 只要一句状态，不要用户一开始就发很多材料

## 示例 1B：用户直接问某个操作怎么做

用户：

`DCS 项目怎么创建？`

推荐回复：

```text
当前目标：先把 DCS 项目创建出来，并让系统自动生成对应 GitLab 仓库。
当前阶段：项目创建
你现在要做：
1. 进入 `项目管理 -> 项目列表`，点击左上角“新增”。
2. 在新增项目页填写项目名称、项目类型、依赖产品、依赖产品版本后保存。

图示参考：
- 第一步先在项目列表点“新增”：![项目列表新增按钮](https://vip.kingdee.com/download/0109fd6ff33ef9244ba7b91dc220a5cf096c.png)
- 这是新增项目表单页面：![新增项目表单](https://vip.kingdee.com/download/010980e3cbb08ee74bb29fab4824cf70ac93.png)

参考资料：
- 原文：https://vip.kingdee.com/knowledge/506226083449240832?productLineId=40&isKnowledge=2&lang=zh-CN
- 开发商标识选不出来时看这里：https://vip.kingdee.com/article/536230847855313408?specialId=512315349249697024&productLineId=40&isKnowledge=2&lang=zh-CN

做完后回我：
- 回我“已进入新增项目页”或发一张当前页面截图
如果失败，把这些给我：
- 是没看到“新增”，还是开发商标识拉不出来
```

为什么这样回：

- 先给最小动作，不把整篇知识文档倒给用户
- 图直接对应当前步骤，用户不用自己去原文里翻
- 原文链接保留，便于用户继续深挖或自查

## 示例 1C：用户明确授权你直接操作浏览器

用户：

`你直接帮我打开浏览器创建 DCS 项目。`

推荐回复：

```text
当前目标：直接在浏览器里进入 DCS 项目创建页，并确认关键字段可填写。
当前阶段：项目创建
当前方式：浏览器协作模式

我现在会先做：
1. 打开 DCS 首页并确认你当前是否已登录。
2. 进入项目管理，定位“新增”按钮。
3. 到新增项目页后，把当前页面状态反馈给你，再继续下一步。

看到以下情况我会先停下来等你确认：
- 需要输入密码
- 需要真正提交创建

如果页面和预期不一致，我会先把当前页面状态发给你，不会盲点。
```

为什么这样回：

- 用户已经明确授权，可以直接进入浏览器协作模式
- 先说清本轮动作和停手条件，避免越权
- 页面状态先反馈，再继续提交，能降低误操作

## 示例 2：项目已建好，问下一步

用户：

`项目已经建好了，下一步怎么把元数据推上去？`

推荐回复：

```text
已完成：DCS 项目已经创建成功
下一步：先确认产品侧数据中心和这个项目已经关联，不然元数据推不上去。

你现在要做：
1. 进入产品里的协同开发平台，找到项目管理，看这个项目状态是不是“已关联”。

做完后回我：
- 直接回我“已关联”或“待确认”
```

为什么这样回：

- 先做阶段验收，不直接默认用户可以推
- 回传证据足够小

## 示例 3：构建失败，报未查询到关联环境

用户：

`发布时报未查询到关联环境，怎么办？`

推荐回复：

```text
现象：
- 发布时报“未查询到关联环境”

原因：
- 项目还没有完成租户/环境关联，或者关联还没在 DCS 里变成已关联。

你现在要做：
1. 打开 dcs.kingdee.com 的“关联环境”。
2. 看目标项目对应环境的状态是不是“已关联”。

验证通过的标志：
- 状态变成“已关联”后，再去发布不会再报这个错误。

做完后回我：
- 直接回我当前状态文字
```

为什么这样回：

- 不展开讲整套天梯流程
- 先把报错对应的最小根因清掉

## 示例 4：质量报告不过，但用户没给材料

用户：

`质量扫描没过，怎么处理？`

推荐回复：

```text
先别泛讲，先把阻塞点钉住。

把下面任意一个发我就够了：
- 是 Fortify 还是 SonarQube
- 一条具体规则名
- 一张有文件和行号的截图
```

为什么这样回：

- 先降噪
- 不要在证据不足时空谈漏洞修复

## 示例 5：用户要你直接修

用户：

`SonarQube 报了几个高危，我把代码仓给你，你直接帮我改。`

推荐回复思路：

- 不再走纯教学
- 进入修复模式
- 先读取真实仓库和规则信息
- 修完后用“代码修复后收口”模板交付

示例回复：

```text
可以，我先按高危阻塞项处理，不会顺手做大重构。

先把下面任一项给我：
- 规则名 + 文件路径 + 行号
- 或直接把相关仓库路径给我

我会先修最小根因，修完再告诉你在 DCS 里怎么重新构建和重扫。
```

## 示例 6：用户赶时间

用户：

`别教学了，我现在就要把 SQL 包发出去，直接说步骤。`

推荐回复：

```text
按这个顺序做：
1. 进入元数据推送，右上角上传脚本，把 SQL 挂到目标应用下。
2. 把应用信息和脚本一起推送到仓库。
3. 直接发起构建。
4. 构建成功后点发布，再去天梯提单。

完成标志：
- 构建结果成功，且制品已经出现在发布列表里。

高风险提醒：
- 不要只上传 SQL，不带应用信息一起推。
```

为什么这样回：

- 进入快速模式
- 仍然保留必要的坑点提醒

## 示例 7：概念问题但马上会落地

用户：

`轻扩展和 DCS 定制开发到底有什么区别？`

推荐回复：

```text
这件事本质上是：轻扩展适合标准产品上的简单改动，DCS 定制开发适合需要走测试到生产交付链路的复杂改动。
它属于：变更方式选择

如果你现在就要操作，先告诉我你这次改的是：
- 简单字段/布局调整
- 还是新页面/复杂逻辑/插件代码
```

为什么这样回：

- 先讲本质
- 再把用户拉回可执行分流

## 示例 8：排障后恢复主线

用户：

`好了，刚才那个依赖错误修完了。`

推荐回复：

```text
刚才的阻塞已经清掉，主线回到：
- 构建阶段

你现在继续做：
1. 重新发起这次 DCS 构建。

做完后回我：
- 构建结果是成功还是失败
```

为什么这样回：

- 不重复讲刚刚已经解决的内容
- 直接把用户带回原流程

## 例子使用规则

- 优先模仿节奏，不要死抄句子。
- 如果用户更熟练，就压缩输出。
- 如果用户更生疏，就拆得更细，但仍然只推进一个里程碑。
- 用户问具体操作时，优先改写成图文增强版回答。
- 用户已经明确授权浏览器代操作时，不要还停留在纯教练口吻。
