* * I. The sandbox or production environment of the ladder indicates that the deployment was successful, but the extended metadata was not found in the development platform. * *
![image](https://vip.kingdee.com/download/0109fba4863849b44ce48586141fb607bb56.png)
* * Analysis: * *

1. View the deployment log of the ladder, search for the lost application code in the log by searching for keywords, and locate the application deployment information "No metadata file under application xxx, exit the deployment metadata".

2. Download the patch package deployed by the ladder and check whether the "application code .xml file" is included under the metadata folder of the application. The patch package in the following figure is missing the xml file named after the application code.Missing this file will cause the system to fail to find the installation path for the metadata and report an error.
![image](https://vip.kingdee.com/download/0109a348011843be4a22850e3cd784287cc1.png)
* * Workaround * *: Re-enter the development platform export application, submit the metadata and other files to the project's repository through the metadata submission tool, and rebuild the patch package for pushing.The point is that after submitting metadata using the Metadata Submit tool, go to the repository and check if the App Encoding .xml file already exists.

Here are the steps to check:

1. Go to the project replication repository address and open it in your browser
![image](https://vip.kingdee.com/download/0109e92b5f30090640499d0b305b76e3c57a.png)
2. Log in with your Git email and password
![image](https://vip.kingdee.com/download/0109914403287d8c4957bbed70fc73f14b0c.png)
3. Enter the datamodel folder of the repository to apply the encoding and name whether there is an xml file to apply the encoding and name under the folder.If so, the issue has been resolved.
![image](https://vip.kingdee.com/download/0109892e59cf2e0f4a6bbe8945ca67d03cb9.png)
* * II. App encoding conflicts * *

* * 1. Problem analysis * *

This error usually occurs when two new applications with the same code are created in the test environment and the same application is expanded in the production environment, resulting in the application of two environments with the same code and inconsistent ID. When the metadata package is deployed in production, the ID in the metadata package does not match the corresponding metadata in the environment, so it is considered that the new operation needs to be created, and the uniqueness constraint of fnumber coding is violated when it is warehoused (see the figure below).

Go to the [app or page metadata conflict analysis] (/tolink? target = https %3A % 2F % 2Fwww.douyin.com % 2Fvideo % 2F7402301094279400743% 3Fprevious_page % 3Dapp_code_link) video.
![image](https://vip.kingdee.com/download/010994592ec3d2994fcd9051d0faae1fc404.png)
* * 2. Solution * *
At the heart of solving this problem is dealing with conflict:

Method: Synchronize the production application to the test environment (recommended first, to ensure that the production content is not lost)
            1) Export the application of the test environment for backup, and delete the application of the test environment
            2) Import the same coded application of the production environment, deploy or import it into the test environment
            3) Develop based on production imported version in test environment
            4) Export and redeploy the content developed in the test environment to the production environment

            4) Export and redeploy the content developed in the test environment to the production environment
