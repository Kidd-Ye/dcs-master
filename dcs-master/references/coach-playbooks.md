# DCS Coach Playbooks

当用户说“带我一步步做”“从零开始”“像教练一样教我”“我不熟悉 DCS”时，优先加载本文件。

输出时优先套用 `response-templates.md`，阶段验收时对照 `milestone-checklists.md`。

## 教练总规则

1. 默认一次只推进一个里程碑，不跨阶段跳讲。
2. 每一轮都输出：
   - 当前目标
   - 当前阶段
   - 你现在要做
   - 做完后回我什么
3. 如果这一步经常出错，再补一行：
   - 如果失败，把什么截图、状态、日志发我
4. 用户一旦发来真实错误、截图、质量报告、仓库内容，立即切换到 `troubleshooting.md` 或代码修复，再回到原路线。
5. 带练时优先使用平台真实路径和按钮名，不要只讲概念。
6. 每过一个里程碑，都做一次 2 行回顾：
   - 已完成：
   - 下一步：
7. 如果用户说“做完了”，不要马上跳下一步；先用 `milestone-checklists.md` 判断该里程碑是否真的完成。

## 开局分流

如果用户没有说明自己在哪个阶段，优先判断属于哪类：

- A. 还没有 DCS 账号 / GitLab 账号
- B. 有账号，但不会创建项目
- C. 项目已经有了，不会推元数据和构建
- D. 要做插件代码开发
- E. 构建失败 / 发布失败 / 部署失败
- F. 质量扫描不过，想修漏洞
- G. 需要接 PCS、翻译平台、应用市场

如果仍不清楚，只问一个问题：

`你现在最接近哪一步：建项目、关联环境、推元数据、构建、看质量报告、还是发布部署？`

## 剧本 A：从零到第一次成功发布

这是最完整的带练路径，适合新手。

### 里程碑 A1：账号与 GitLab

- 目标：能登录 DCS，并在产品侧 `项目管理` 里完成 GitLab 登录。
- 要做：
  - 登录 `dcs.kingdee.com`
  - 绑定常用邮箱
  - 通过邮件设置 GitLab 密码
- 完成信号：
  - DCS 可正常登录
  - 产品侧 `项目管理` 不再卡在 GitLab 登录
- 常见卡点：
  - 没收到设置密码邮件
  - Git 密码忘记，需要重置

### 里程碑 A2：开发商标识

- 目标：在建项目时能选出正确的开发商标识。
- 要做：
  - 在开发者门户创建或确认开发商标识
  - 确认组织类型与项目类型匹配
  - 确认当前操作人已加入开发商团队
- 完成信号：
  - 创建项目时能正常选出开发商标识
- 常见卡点：
  - 定制化项目误用 ISV 标识
  - 用户不是开发商团队成员

### 里程碑 A3：创建项目与邀请成员

- 目标：DCS 项目建好，仓库自动生成，团队成员可加入。
- 要做：
  - 填项目类型、依赖产品、依赖产品版本
  - 保存项目，确认自动生成仓库
  - 邀请成员并完成审核
- 完成信号：
  - DCS 中能看到项目、成员列表、仓库入口
- 常见卡点：
  - 把“一次构建”误以为要建“一个新项目”
  - 只加了项目成员，没加开发商团队成员

### 里程碑 A4：关联数据中心与环境

- 目标：产品侧能关联项目，DCS 侧能关联租户/环境。
- 要做：
  - 如果从产品推元数据，先做数据中心关联
  - 如果要发布，先做租户或私有云环境关联
- 完成信号：
  - 状态变成 `已关联`
- 常见卡点：
  - 产品侧项目看不到
  - 发布时报 `未查询到关联环境`

### 里程碑 A5：把元数据或代码送进仓库

- 目标：仓库里已经有这次构建所需的真实内容。
- 要做：
  - 推送元数据，或者提交插件代码，或者上传 SQL/静态资源
  - 如果推页面，记得把应用一起推
- 完成信号：
  - 仓库能看到对应 `datamodel`/`code`/`webapp` 内容
- 常见卡点：
  - `数据中心版本低于仓库版本`
  - `元数据的id不一致`
  - 仓库 `datamodel` 结构不完整

### 里程碑 A6：构建

- 目标：得到一个可发布的 DCS 制品。
- 要做：
  - 选正确的构建类型：元数据、元数据+插件、插件代码、部分包、SQL、静态资源
  - 必要时选合适的构建依赖版本
- 完成信号：
  - 构建结果为成功
- 常见卡点：
  - `gradle.properties不存在`
  - `依赖错误`
  - `datamodel 内容为空`

### 里程碑 A7：质量/测试报告

- 目标：制品达到可发布状态。
- 要做：
  - 如果包含代码，等待质量扫描
  - 如果要求测试报告，填写、提交、审核
- 完成信号：
  - 质量扫描通过
  - 测试报告通过
- 常见卡点：
  - Fortify / SonarQube 不通过
  - PCS 未关联导致需求清单无法导入

### 里程碑 A8：发布与部署

- 目标：制品成功发布到天梯或私有云平台，并完成验证。
- 要做：
  - 在 DCS 或产品侧发布
  - 在天梯提单
  - 先验证沙箱，再申请生产
- 完成信号：
  - 天梯任务成功，业务验证通过
- 常见卡点：
  - 风险等级 C/D 需要审核
  - 部署成功但目标环境里看不到元数据

## 剧本 B：已有项目，第一次推元数据并构建

适用于“账号、项目都好了，现在想第一次走通构建”。

建议按这 5 步带：

1. 确认产品侧项目关联状态
2. 确认仓库里是否已有正确的 `datamodel`
3. 指导用户推送这次变更的元数据
4. 帮用户选构建类型
5. 构建成功后，判断是否需要质量扫描和测试报告

每一轮都要让用户反馈一个明确结果，例如：

- `项目状态是否已关联`
- `仓库里是否看到了应用和页面`
- `构建结果是否成功`

是否推进下一步，以 `milestone-checklists.md` 的完成定义为准。

## 剧本 C：插件开发专项

适用于“我要做插件代码，不只是元数据”。

带练顺序：

1. 安装 IDEA 小助手
2. 用仓库地址初始化 Gradle 工程
3. 首次初始化后，把 DCS 云构建需要的文件完整提交
4. 跑通本地 debug
5. 提交代码并在 DCS 发起代码相关构建
6. 如果构建失败，切到 `troubleshooting.md`

重点提醒：

- 不要把 IDEA 本地专用的 Gradle 简化配置误提交到 DCS 仓库
- DCS 会解析 repo 里的 Gradle 结构，尤其是 `all` 工程

## 剧本 D：质量扫描与漏洞修复专项

适用于“构建成功了，但质量不通过”。

带练顺序：

1. 先分清是 Fortify 还是 SonarQube
2. 优先看严重级别和高危级别
3. 定位到文件和行号
4. 如果用户要你直接修，就在真实仓库里修
5. 修完后明确告诉用户如何重新构建和重新扫描

每轮只推进一个缺陷批次，优先：

- 严重
- 高危
- 中危

不要把所有低优先级建议和当前阻塞问题混在一起。
输出时优先用 `response-templates.md` 的“排障输出”和“代码修复后收口”模板。

## 剧本 E：构建/发布/部署救火

适用于“现在卡住了，先帮我恢复流程”。

建议按这个顺序排：

1. 是关联问题，还是仓库问题，还是构建问题，还是部署问题
2. 先确认最前置的阻塞是否已解除
3. 修最小根因
4. 验证是否恢复
5. 恢复后，告诉用户应该回到哪一步继续

典型切入词：

- `未查询到关联环境`
- `数据中心版本低于仓库版本`
- `元数据的id不一致`
- `gradle.properties不存在`
- `依赖错误`
- `datamodel内容为空`
- `部署成功但环境里找不到页面/应用`

## 剧本 F：周边集成

适用于 PCS、翻译平台、应用市场这三类。

- PCS：主要影响测试报告需求清单导入
- 翻译平台：主要影响多语言补丁进入 DCS 制品列表再发布
- 应用市场：主要影响应用关联、加签、生态产品构建

这三类都不要混入主构建链路一起讲，除非用户当前就在做这件事。

## 建议输出长度

- 新手带练：优先短回合，3 到 8 行正文即可
- 排障：优先结构化，通常按“现象/原因/修复/验证”
- 代码修复：先给结论，再给变更点，再给 DCS 验证步骤
